Troubleshooting Notes
Branch Office Site-to-Site IPsec VPN Lab


Lab Goal
The goal of this lab was to configure and verify a site-to-site IPsec VPN between two branch office networks.

Site A LAN: 192.168.10.0/24
Site B LAN: 192.168.20.0/24

R1 WAN IP: 10.0.0.1
R2 WAN IP: 20.0.0.2


Network Summary
Item	Value
VPN Type	Site-to-Site IPsec VPN
Site A Network	192.168.10.0/24
Site B Network	192.168.20.0/24
R1 WAN IP	10.0.0.1
R2 WAN IP	20.0.0.2
Authentication	Pre-shared key
Verification	Ping, ISAKMP, IPsec counters


Troubleshooting Scenarios
Scenario 1: PC-A Could Not Ping PC-B Before VPN Symptom
PC-A could not ping PC-B.


Test command:
ping 192.168.20.10


Result:
Request timed out


Cause
The site-to-site VPN had not been configured yet.
The ISP router also did not have routes to the private LAN networks.


Fix
Configured IPsec VPN between R1 and R2.
Configured interesting traffic ACLs on both routers.
Configured static routes on R1 and R2 to reach the remote LANs.


Result
After VPN configuration, PC-A successfully pinged PC-B.


Scenario 2: Static Routes Were Missing on R1 and R2 Symptom
R1 and R2 did not show routes to the remote LAN networks.


Checked with:
show ip route


Cause
Static routes to the opposite site LAN were missing.


Fix
Added this route on R1:
ip route 192.168.20.0 255.255.255.0 10.0.0.2


Added this route on R2:
ip route 192.168.10.0 255.255.255.0 20.0.0.1


Verification
On R1:
show ip route


Expected result:
S 192.168.20.0/24 via 10.0.0.2


On R2:
show ip route


Expected result:
S 192.168.10.0/24 via 20.0.0.1


Scenario 3: VPN Tunnel Did Not Come Up Immediately Symptom
After configuring the VPN, the tunnel did not show as active immediately.


Checked with:
show crypto isakmp sa


Cause
No interesting traffic had been generated yet.
IPsec VPN tunnels usually come up only when traffic matches the VPN ACL.


Fix
Generated traffic from PC-A to PC-B:
ping 192.168.20.10


Result
After sending traffic, the VPN tunnel came up.


Verified with:
show crypto isakmp sa


Expected result:
QM_IDLE


Scenario 4: First Ping Failed Symptom
The first ping from PC-A to PC-B failed.


Cause
The VPN tunnel was still negotiating.
This is normal behavior for IPsec VPNs.


Fix
Ran the ping command again:
ping 192.168.20.10


Result
The second ping succeeded.


Scenario 5: IPsec Counters Needed Verification Symptom
Basic ping worked, but VPN encryption needed to be verified.


Verification Command
On R1 and R2:
show crypto ipsec sa


What I Checked
I checked that these counters increased:
packets encaps
packets decaps


Meaning
Counter	Meaning
packets encaps	Packets encrypted by the router
packets decaps	Packets decrypted by the router
Result
The counters increased after traffic was sent between PC-A and PC-B.
This confirmed that traffic was encrypted through the VPN tunnel.
